#NPM学习笔记

1. 常用的命令

  • npm list 列出当前目录下的npm安装的package
  • npm config get [key] 获取npm配置文件中指定key的value
  • npm install [package-name] 安装指定的包
  • npm update 升级package(该命令要在需要升级的包下运行)
  • npm uninstall [package-name] 卸载包
  • npm uninstall --save [package-name] 卸载依赖

注:

  • 可以使用npm [tab][tab]显示出所有的命令，然后npm [命令] --help显示该命令的具体的用法
  • 使用-g参数是对全局生效

2. npm版本管理

  • 版本格式: 主版本号.次版本号.修订号
  • 版本递增规则:
    1. 主版本号: 做了不兼容的API修改
    2. 次版本号: 做了向下兼容的功能性新增
    3. 修订号: 做了向下兼容的问题修正
  • 额外: 先行版本号及版本编译信息可以加到”主版本号.次版本号.修订号”的后面,作为延伸.

3. 具体使用

  • *: 任意版本
  • 1.1.0: 指定版本
  • ~1.1.0: >=1.1.0 && < 1.2.0 (用于指定次版本, 此版本 + 1)
  • ^1.1.0: >=1.1.0 && < 2.0.0 (用于指定主版本, 主版本 + 1)

4. npm常见的问题

  • npm install 安装的包使用npm list出现extraneous的包, 是因为包没有出现在package.json的依赖中, 可以使用npm-prune移除没有关联的包

2. ApplicationContext 和 WebApplicationContext

  • 区别: ServletContext 是web应用程序的大环境,用于存储整个web应用程序级别的对象,由spring管理的对象的Map结构就是存储在ServletContext中的.
    ApplicationContext,WebApplicationContext 是Spring的BeanFactory,一个是支持web特性的BeanFactory.

  • Spring获取ApplicationContext和WebApplication的方法

    1. 在初始化时保存ApplicationContext对象

    2. 通过Spring提供的工具类获取ApplicationContext对象

      说明: 这种方式适合于采用Spring框架的B/S系统,通过ServletContext对象获取ApplicationContext对象,然后在通过它获取需要的类实例.
      上面两个工具方式的区别是,前者在获取失败时抛出异常,后者返回null.
      其中 servletContext sc 可以具体 换成 servlet.getServletContext()或者 this.getServletContext() 或者 request.getSession().getServletContext(); 另外，由于spring将注入的对象放在ServletContext中的，所以可以直接在ServletContext取出 WebApplicationContext 对象:
      WebApplicationContext webApplicationContext = (WebApplicationContext) servletContext.getAttribute(WebApplicationContext.ROOT_WEB_APPLICATION_CONTEXT_ATTRIBUTE);

    3. 继承自抽象类ApplicationObjectSupport
      说明: 抽象类ApplicationObjectSupport提供getApplicationContext()方法,可以方便的获取到ApplicationContext.
      Spring初始化时,会通过该抽象类的setApplicationContext(ApplicationContext context)方法将ApplicationContext对象注入.

    4. 继承自抽象类WebApplicationObjectSupport
      说明: 类似方法3,调用getWebApplicationContext()获取WebApplicationContext

    5. 实现接口ApplicationContextAware
      说明: 实现该接口的setApplicationContext(ApplicationContext context)方法,并保存ApplicationContext 对象.Spring初始化时，会通过该方法将ApplicationContext对象注入.

###　4. DispatcherServlet处理请求的顺序

  • WebApplicationContext作为一个属性将被绑定到请求中, 相应的属性名:DispatcherServlet.WEB_APPLICATION_CONTEXT_ATTRIBUTE, 这样在整个请求过程中的Controller和其他元素就可以使用WebApplicationContext

  • 返回模型, 渲染视图, 也有可能没有模型返回, 因为有可能是preprocessor或者 postprocessor拦截了请求
  • Handler exception resolvers 用于处理请求出错时返回的视图
  • DispatcherServlet也支持Last-Modify, 只要Controller方法实现LastModify接口实现getLastModified方法就可以了, 具体参考DispatcherServlet实现Last-Modify

  • 添加DispatcherServlet的初始化参数(init-param)

    |可选参数|含义|
    |:——:|:——-:|
    |contextClass|WebApplication的实现类, 即spring mvc的上下文对象|
    |contextConfigLocation|spring mvc的配置文件, 该值会被传入到WebApplication的实现类中, 支持配置多个配置文件(以逗号分隔), 如果配置文件之间存在重复定义的bean, 则以最后的覆盖之前的|
    |namespace|WebApplication的命名空间, 默认是[servlet-name]-servlet|

#ClassLoader

####1. 什么是classloader？

  • ClassLoader就是用来动态加载class文件到内存中的类

####2. Java提供的默认的三个ClassLoader

  • BootStrap ClassLoader: 启动类加载器，负责加载jdk中的核心类库，如rt.jar，resources.jar，charsets.jar等，可以通过下面的程序获得该类加载器从哪些地方加载了相关的jar或者class文件

运行结果：

其实上述结果也是通过查找sun.boot.class.path这个系统属性所得知的。

  • Extension ClassLoader: 扩展类加载器，负责加载Java的扩展类库，默认加载JAVA_HOME/jre/lib/ext/目下的所有jar。

  • App ClassLoader: 系统类加载器，负责加载应用程序classpath目录下的所有jar和class文件。

3. ClassLoader加载类的原理

  • ClassLoader使用的是双亲委托模型来搜索类的，每个ClassLoader实例都有一个父类加载器的引用（不是继承的关系，是一个包含的关系），虚拟机内置的类加载器（Bootstrap ClassLoader）本身没有父类加载器，但可以用作其它ClassLoader实例的的父类加载器。当一个ClassLoader实例需要加载某个类时，它会试图亲自搜索某个类之前，先把这个任务委托给它的父类加载器，这个过程是由上至下依次检查的，首先由最顶层的类加载器Bootstrap ClassLoader试图加载，如果没加载到，则把任务转交给Extension ClassLoader试图加载，如果也没加载到，则转交给App ClassLoader 进行加载，如果它也没有加载得到的话，则返回给委托的发起者，由它到指定的文件系统或网络等URL中加载该类。如果它们都没有加载到这个类时，则抛出ClassNotFoundException异常。否则将这个找到的类生成一个类的定义，并将它加载到内存当中，最后返回这个类在内存中的Class实例对象。

  • 使用双亲委托模型的原因: 这样可以避免重复加载，当父亲已经加载了该类的时候，就没有必要子ClassLoader再加载一次。考虑到安全因素，我们试想一下，如果不使用这种委托模式，那我们就可以随时使用自定义的String来动态替代java核心api中定义的类型，这样会存在非常大的安全隐患，而双亲委托的方式，就可以避免这种情况，因为String已经在启动时就被引导类加载器（Bootstrcp ClassLoader）加载，所以用户自定义的ClassLoader永远也无法加载一个自己写的String，除非你改变JDK中ClassLoader搜索类的默认算法。

  • JVM判断两个class是否相同的原理: JVM在判定两个class是否相同时，不仅要判断两个类名是否相同，而且要判断是否由同一个类加载器实例加载的。只有两者同时满足的情况下，JVM才认为这两个class是相同的。就算两个class是同一份class字节码，如果被两个不同的ClassLoader实例所加载，JVM也会认为它们是两个不同class。

1. 函数接口

  • 函数接口是只有一个抽象方法的接口， 用作lambda表达式的类型
  • 如果要将lambda表达式定义成一个接口类型，则要将lambda表达式的类型和该接口中的唯一一个抽象方法的签名保持一致
1.1 jdk提供的常用的函数接口
接口 参数 返回类型
Predicate<T> T boolean
Consumer<T> T void
Function<T> T R
Supplier<T> None T
UnaryOperator<T> T T
BinaryOperator<T> (T,T) T

2. 惰性求值方法/及早求值方法

  • 只描述Stream最终不产生新的集合的方法叫做惰性求值方法, 例如filter方法
  • 最终会从Stream产生值的方法叫做及早求值方法, 例如count方法

注意: 判断一个操作是惰性求值还是及早求值只需要看它的返回值, Stream->惰性求值, void/另一个具体的值->及早求值

3. 注意点

  • lambda表达式中不能调用接口中的默认方法
  • 可以使用reduce对stream中的两个元素同时操作
