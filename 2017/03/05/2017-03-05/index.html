Clean Code读书笔记1 -- 命名和函数

Posted on 2017-03-05   |   In Clean Code   |  

Reading notes for Clean Code

1. 命名

众所周知, 编程中最难的两件事情, 一个是缓存, 另一个就是命名. 一个好的命名可以省去很多的理解成本, 一个不好的命名可能与要描述的事实背道而驰, 由此可见一个好的命名是多么的重要.

一个好的命名需要遵守的原则包括但不仅限于以下几点

  • 名副其实

    一个名字要表达的意思应该就是它的字面意思, 而不需要用注释进行解释, 否则这样的命名就是存在问题的.

  • 做有意义的区分

    要给名字这个字段取变量名应该直接用name, 而不需要是nameString, 加上String作为限定没有任何的意思. 在我们自己的代码中有的时候会这样取名

    其实加上List作限定没要必要, 可以直接取名为 List<Doctor> doctors

  • 使用可以搜索的名称

    这个也可以描述成不要在代码中存在魔数, 应该使用常量跟踪.
    例如: 使用WORK_DAYS_PER_WEEK来替代魔数5.

  • 不需要使用成员前缀来标明成员变量

    例如对于一个成员变量, 不需要在变量名前面加上m_来表示

  • 类名和对象名应该是名词或者是名词短语, 不应该是动词

  • 方法名应该是动词或者是动词短语.

  • 每个概念对应一个词

    在特定的业务中可能有多个词汇可以表示同一个意思, 但是要选定一个词汇来表示并且以后都使用同一个词汇来表示
    例如: 杏仁号就是使用serial来表示, 而不应该是在privdoc是用serial, 在clinic是使用xingrenNo.

2. 函数

函数可以说是一个程序中最重要的组件, 写代码可以说很大一部分时间是在编写函数, 因此函数编写的好坏直接决定了代码编写的好坏

  • 短小

    函数需要短小再短小, 精炼再精炼, 如果一个函数就有几百行, 那就说明这个函数功能太复杂, 职责太多, 干了太多的事情, 抽象程度不够.

  • 只做一件事

    函数应该只做一件事情. 也就是所谓的单一原则(SRP). 这个原则和上一条原则也是相互补充的, 如果一个函数只做一件事情那这个函数应该是短小的, 反之就有可能做了不止一件事情.

  • 自顶向下规则

    函数调用者和被调用者之间的距离应该尽可能的小, 而不应该根据类别进行划分, 这样在阅读代码的时候可以快速的在函数之间查看.
    在我们自己的代码中之前是将一个类中的私有方法放到类的最下面和其他的方法进行区分, 经过讨论之后决定采用自顶向下原则, 将私有方法和其他方法同等对待, 放在调用者的下面.

  • switch语句

    switch语句的出现一般就表示这个地方以后可能会需要更改了, 而且可能有其他的一些地方会用到相同的switch语句. 因此switch语句的出现就要考虑是否需要使用工厂模式了, 将变化控制在一个地方而不是分散到系统的各个角落中.

  • 函数参数

    一个函数的参数需要进行控制, 不能无限制的增长, 如果一个函数的参数太多, 对于这个函数的调用者来说也是很不友好的. 一般函数的参数以3个以下为宜, 如果再多可以考虑封装成一个对象.

  • 无副作用

    一个函数中做的事情就应该仅仅是函数名中提及到的事情, 如果函数在承诺只做一件事情的前提下又去更新其他的东西并且这个更新操作用户是不知情的, 那么这个函数就违背了SRP原则并且产生了副作用. 要避免这样的情况发生.

  • 别重复自己

    编写一个良好的函数的原则之一就是不要写重复的代码, 即DRY原则, 一旦违反了DRY原则就意味着以后要修改的时候可能就要修改很多处地方, 而且不能遗漏, 一旦遗漏一处就会造成bug的产生.

  • 分离指令与询问

    即CQS原则. 也就是说函数要么做什么事情, 要么回答什么事情, 但是不能同时做两件事情.
