Reading notes for Clean Code – serial 2

1. 对象和数据结构

  • 对象和数据结构的定义和不同

    对象的职责是把数据隐藏于抽象之后, 暴露操作数据的接口, 使外界不直接操作其内部的数据.
    数据结构是暴露其数据, 不提供有意义的函数.
    两者的职责和功能要分清

  • 面向过程式的代码和面向对象式的代码

    但是面向过程式的代码在添加新的函数的时候可以不用修改原来的数据结构, 例如在Geometry类中添加一个新的函数的时候, 其他的数据结构不需要进行任何的修改.

    可以看到, 使用面向对象式的代码, 如果要添加一个新的数据结构, 只要实现Shape接口就可以了, 原本的对象不需要进行任何的改变.
    但是, 如果要添加新的函数(行为), 例如在Shape接口中添加一个新的函数, 那么其他几个实现类都需要进行修改.

  • Demeter法则

    模块不需要了解其所操作对象的内部情形(具体的数据结构), 而只需要操作对象暴露的接口即可.
    Demeter法则任务, 类C的方法f值应该调用以下对象的方法:

    • 类C中的方法
    • 由f创建的对象中的方法
    • f参数中的对象方法
    • C中成员对象中的方法

2. 边界

一个项目中的边界有很多, 第三放代码, 其他人未实现的代码等都是项目的边界.

  • 浏览和学习边界

    这里涉及到的边界是指程序中的第三方代码. 在实现自己的程序的时候秉持不重复造轮子的原则, 或多或少都会涉及到第三方代码, 我们应该要事先考虑第三方代码和自己的代码之间的关系, 和第三方代码在自己项目中的渗透度.

    在使用第三方代码的时候, 首先要对边界有一定的了解. 因此有必要对边界进行学习. 学习性测试是了解和学习边界的一个有效的办法. 我们可以在将第三方代码用到项目中之前先编写测试用例来学习第三方代码, 在测试用例中对要在真正代码中使用的功能进行调用, 也可以进入第三方代码中, 研究其具体实现和原理, 在对第三方代码有比较深入的了解之后如果以后在真正代码中使用时遇到问题也可以快速的解决问题.

    使用学习性测试的另外一个好处是当第三放代码升级之后, 可以运行之前的测试用例, 看看哪些功能被改变了, 评估升级的风险.

  • 使用尚不存在的代码

    这里涉及到的边界就是其他人未实现的代码. 在一个项目开发过程中, 每个模块之间肯定需要进行交互, 模块和模块之间势必存在依赖关系. 如果自己开发的模块所依赖的模块还没有就绪的时候该怎么办?

    我们可以自己先实现对应的模块, 但是自己实现模块的返回值都是一个fake的值, 这样我们自己的模块就可以使用这个fake的模块继续进行开发了. 等真正的模块被提供的时候再将其替换成真正的模块. 因为fake的模块和真正的模块提供的接口可能不会一模一样, 所以可以使用Adapter模式接入真正的模块.

  • 关于第三方类库的一点总结

    在使用第三方代码的时候, 可以在第三方代码之上封装一层而不是之间使用第三方代码, 因为如果这个第三方代码在自己的代码中渗透的比较广泛之后, 如果要对第三方代码进行升级而且升级之后改动较大就会造成要修改很多地方的情况. 如果是在第三方代码之上封装了一层, 在程序中使用的都是我们封装之后代码, 那升级的时候也只需要修改我们封装的代码就可以了. 也就是说升级对于项目本身是透明的.
